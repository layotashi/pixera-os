# SYNESTA — Product Brief

> このドキュメントは SYNESTA の製品コンセプト・対象ユーザー・差別化要因・設計原則を
> 1 箇所にまとめた **living document** です。
> ランディングページ・SNS・README (公開版)・ピッチ資料など、
> すべての対外コミュニケーションの **源泉** として機能します。

---

## 1. Vision

> **電子情報技術の高度化により失われた「こだわり切れる」美学を、
> 空想のレトロマシンとして現代に蘇らせる。**

現代のコンピューティングは高度化の果てに「アナログと見分けがつかないデジタル」に到達した。
音楽も映像もすべての表現がアナログ的曖昧さを纏い、人間が認知できないレベルの複雑さゆえに
"こだわり切れない" もどかしさが生まれている。

SYNESTA は、制約の中でこそ到達できる **完璧なデザイン** ——画面の 1 ピクセル、
音の 1 波形まで意図どおりに支配できる体験——をクリエイターに届ける。

ただしそれは歴史的名機のエミュレーションではなく、**空想のマシン** としての再解釈である。
SNS 共有のしやすさや "映える" 制作画面といった現代的な必須機能を多分に内包しながら、
レトロを愛好するクリエイターが制作の隅々まで満足感を得られる環境を目指す。

## 2. Product Concept

SYNESTA は、目の前に立ち上がる、2 色だけの空想のレトロコンピュータです。
起動した瞬間から空想のマシンの中にいて、作ることも、その画面を眺めることも、
ひとつの一貫した美学のもとで起きる。高度化の果てに曖昧になった現代のデジタルとは逆に、
ここでは細部の一つひとつを自分の意図で支配できます。

> 一行の要約は `config.js` の `APP_DESCRIPTION` が正典 (ABOUT ダイアログに表示)。

### 2.1 なぜ 1-bit なのか

- **レトロ PC への美的共感**: 初期 Macintosh やゲームボーイなど、
  限られた表示性能の中で生まれた明快で力強いビジュアルへの敬意。
- **技術的シンプルさ**: レンダリングパイプラインを極限まで単純化することで、
  VRAM の 1 ピクセル単位まで完全に意図どおりに制御できる。
  「色が 2 色しかない」ことで、形・構成・レイアウトという本質的なデザイン要素だけが残る。
- **「こだわり切れる」複雑さ**: 高度に複雑なシステムでは、人間の認知を超えた要素を
  妥協せざるを得ない。1-bit の制約は、制作者が全体を把握し、
  すべてのディテールに意図を込められる規模にシステムを留める。

> **補足**: 今後のプロトタイピング次第で、2-bit (明暗 4 階層) や
> モノクロ＋アクセントカラーへの拡張を検討する可能性はある。
> ただしそれ以上（256 色等）への緩和は行わない——制約の解除はコンセプトの喪失を意味する。

### 2.2 なぜ "デスクトップ環境ごと" 作るのか

| 動機                       | 説明                                                                                                                                  |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| **没入感・世界観の構築**   | ツールを起動した瞬間から空想のレトロマシンの中にいる。単一アプリの起動ではなく「OS にログインする体験」がある。                       |
| **制作ワークフローの統合** | DAW・ペイント・メモ帳など、クリエイターが必要とする機能群を同一美学のもとに統合する。外部ツールへの行き来によって世界観が途切れない。 |
| **製品としての独自性**     | DAW 単体では既存製品との機能比較で埋もれる。「OS ごと体験する」という次元の違いで差別化する。                                         |

今後、LiveCoding・アニメーション・ゲーム制作など、クリエイティブ用途を中心に
機能を拡張していく予定。

### 2.3 なぜブラウザ完結・ビルドレスなのか

当初は Vanilla JS + Canvas API + Web Audio API を **高速プロトタイピング** のために
選択した技術スタックだった。
しかし、ブラウザで動作すること自体が製品の強みになりつつある：

- **インストール不要**: URL を開くだけで体験が始まる。
- **SNS 共有との親和性**: p5.js がジェネラティブアート界隈で独自の共有文化を生んだように、
  ブラウザネイティブであることはコミュニティ形成の土壌になり得る。
- **ゼロ依存**: npm パッケージも bundler もない。コードそのものがプロダクトである。

## 4. Key Differentiators

1. **制作画面 = 作品**
   制作中の画面そのものが映像作品として成立する。録画ボタンを押すだけで
   SNS 共有用コンテンツが完成する。OBS 等の外部キャプチャは不要。

2. **制約が制作フロー全体を貫く**
   アウトプット (チップチューン・ドット絵) だけでなく、制作プロセスそのものが
   1-bit の美学に支配される。「道具」と「作品」の境界が溶ける体験。

3. **空想のマシン**
   実在ハードウェアのエミュレーションではなく、現代的機能を内包した
   架空のレトロコンピュータ。リアルさではなく「美しさ」で設計判断を行う。

4. **URL を開くだけ**
   インストール・アカウント作成・ビルドの一切が不要。
   ブラウザで即座に、完全な制作環境に入れる。

5. **統合されたクリエイティブ OS**
   DAW 単体ではなく、ペイント・メモ帳・ライフゲーム等を含む
   OS 風デスクトップ体験。ツールを使う行為自体が世界観への没入となる。

## 5. Design Principles

開発上の判断に迷ったとき、以下の原則に立ち返る。

1. **美しい再解釈 over 正確なエミュレーション**
   歴史的ハードウェアの仕様忠実再現よりも、それが美しいなら現代的解釈を選ぶ。
   SYNESTA は「空想のマシン」であり、過去の "正解" に縛られない。

2. **映える画面 over 隠れた機能性**
   制作画面そのものが鑑賞に堪えるかを常に問う。
   制作作業に直接関係しない機能 (ライフゲーム、スプラッシュスクリーン等) であっても、
   映像としてのおもしろさがあれば積極的に採用する。

3. **美しさの妥協なし**
   デザイン・配置・ワークフロー・統一感において妥協しない。
   1 ピクセルのズレ、1 要素の不調和も許容しない。
   これが 1-bit で「こだわり切る」ことの本質である。

4. **即触れる体験 over 高度な導入フロー**
   URL を開けば始まる。チュートリアルなしでも「触ってみたい」と思わせる
   画面の引力を優先する。

5. **設計根拠は SYNESTA 内部の原則から導く**
   特定 OS の慣習 (「Windows ではこう」「Mac はこう」「X11 だと…」) を
   設計判断の権威として援用しない。SYNESTA は「空想のマシン」(§1) であり、
   実在 OS の clone でも子孫でもない。

   結果的に既存 OS の慣習と一致する選択をしてもよいが、根拠は SYNESTA 文脈で
   説明する。例:
   - NG: 「Windows のシステムメニューと同じく header 右クリックで開く」
   - OK: 「ヘッダーは窓を代表する場所なので、窓への操作の入口として直感的」

   根拠の源泉は本ドキュメント・`HUMOR_PRINCIPLES.md`・その他プロジェクト内
   ドキュメントとする。

6. **UI の数値には必ず単位を併記する**
   STUDIO の `MS` / `%` / `DEG` を規範として、数値パラメータには常に単位ラベルを表示する。

   | 単位  | 対象                                       | 表記例                                      |
   | ----- | ------------------------------------------ | ------------------------------------------- |
   | `DOT` | VRAM ピクセル数 (SYNESTA 座標系のピクセル) | Header pad, Content pad, Spacing, Thickness |
   | `%`   | 強度・割合                                 | Strength, Radius, Level                     |
   | `MS`  | ミリ秒                                     | ADSR の Attack, Decay, Release              |
   | `DEG` | 角度                                       | Phase                                       |

   **`DOT` について**: SYNESTA の VRAM ピクセルは CSS `px` や物理ピクセルとは次元が異なる。
   `px` と書くと Web の CSS ピクセルと混同されるため、SYNESTA 座標系のピクセルは
   `DOT`（dot — 表示素子 1 つ）と表記する。
   速度・無次元値 (0〜100 スケールのスライダー等) は単位不要。

## 6. Competitive Landscape

| 製品 / プロジェクト       | カテゴリ                     | SYNESTA との違い                                                                                                                                                                                           |
| ------------------------- | ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **LSDJ**                  | ゲームボーイ用トラッカー     | 実機実行・実機互換の魅力があるが、SNS 共有は困難。SYNESTA は共有前提の設計。                                                                                                                               |
| **FamiTracker**           | NES 音源トラッカー           | 実機音源の忠実再現が目的。SYNESTA は忠実さよりも美的統一感を重視。                                                                                                                                         |
| **Beepbox**               | Web ブラウザ・チップチューン | ブラウザ動作の手軽さは近いが、OS 体験・ビジュアル美学という次元がない。                                                                                                                                    |
| **PICO-8**                | Fantasy Console              | 思想的に最も近い。ただし PICO-8 は「仮想ゲーム機 = 携帯ゲーム機文化」であり、SYNESTA は「仮想 OS = レトロコンピュータ・ハッカー文化」。PICO-8 が広い市場を狙うのに対し、SYNESTA はニッチな美的体験に特化。 |
| **TIC-80**                | Fantasy Console (OSS)        | PICO-8 と同カテゴリ。SYNESTA とはメタファー (ゲーム機 vs OS) と用途 (ゲーム開発 vs クリエイティブ制作全般) が異なる。                                                                                      |
| **Ableton Live / Cubase** | プロ向け DAW                 | 機能性は圧倒的だが、制作画面は「機能のための道具」であり鑑賞対象ではない。SYNESTA は制作画面自体を作品化する。                                                                                             |

## 7. Product Boundaries

SYNESTA が **やらないこと** を明示する。

- **歴史的ハードウェアの正確なエミュレーション**: 実機仕様の忠実再現は目的ではない。
  音源チップのサイクル精度再現や、特定機種の制約の正確な模倣は行わない。
- **プロ向け DAW としての機能完備**: Ableton / FL Studio 的な多機能 DAW は目指さない。
  現代的な超マルチトラック構成やVSTi プラグイン等は対象外。
- **256 色以上への表示拡張**: 制約の大幅な緩和はコンセプトの喪失を意味する。
  (2-bit / モノクロ＋アクセントカラー程度の拡張は検討対象)
- **ネイティブアプリ配布**: デスクトップ / モバイルのインストーラ配布はスコープ外。
  ブラウザ完結であることが製品の強み。(必要に応じて検討する可能性はあるが、当面はブラウザ完結で開発を進める)

## 8. Tone & Messaging Guidelines

対外コミュニケーション (ランディングページ・SNS・README) で意識するトーン：

### 採用するトーン

- **洗練・ミニマル**: 言葉も 1-bit のように削ぎ落とす。余計な装飾語を避け、
  短く鋭いコピーを志向する。
- **アンダーグラウンド・カウンターカルチャー**: メインストリームの DAW 文化に対する
  静かな反骨心。「みんなが使うもの」ではなく「わかる人にはわかる」空気感。
- **知的・コンセプチュアル**: 単なるツール紹介ではなく、背景にある思想や美学が
  自然に滲む表現。アートステートメント的な密度。
- **計算されたユーモア**: ジョークプログラム・イースターエッグには遊び心を許容するが、
  構造的な設計原則に基づいて仕込む。詳細は `HUMOR_PRINCIPLES.md` を参照。

### 避ける表現

- **「簡単」「誰でも」の乱用**: 間口は広くするが、安売りはしない。
  制約と向き合う覚悟を持つユーザーに語りかける。
- **機能リストの羅列**: スペック紹介ではなく「体験」を伝える。
- **歴史的名機との直接比較**: 「○○より優れている」ではなく、
  「SYNESTA にしかない体験」を語る。

### SNS 運用指針 (X / building in public)

**基本方針**: 高品質ビジュアル＋最小限の言葉。画像が語り、テキストは補助。

**投稿トーン**:

- キャプションは短く、詩的に。技術仕様の羅列ではなく「体験の断片」を見せる。
- 質問を添える場合は美学的判断 (「A or B?」) に留める。
  機能要望の募集は慎重に——プロダクトの方向性をフォロワーの多数決で決める姿勢は
  SYNESTA のカウンターカルチャー的トーンと矛盾する。

**投稿パターン**:

| パターン       | ビジュアル                           | キャプション例                 |
| -------------- | ------------------------------------ | ------------------------------ |
| 新機能紹介     | ループ GIF / 短動画 (15s 以内)       | 機能の本質を 1 文で + 問いかけ |
| 美学・雰囲気   | デスクトップ全体のスクリーンショット | コンセプトを短く               |
| Before → After | 2 枚組比較                           | 変化点を一言だけ               |

**ハッシュタグ戦略**:

- **アイデンティティタグ** (自ら名乗るもの): `#1bit` `#retrocomputing` `#lowres` `#indiedev`
- **リーチ用タグ** (見つけてもらうためのもの): `#pixelart` `#gamedev` `#ScreenshotSaturday`

**頻度・タイミング**:

- 週 2–3 回。多すぎるとアルゴリズム上逆効果。
- #ScreenshotSaturday: 土曜 20:00–24:00 JST (UTC 11:00–15:00)
- 平日: 20:00–22:00 JST

**バイラルループの意識**:

- 開発者 (自分) の投稿は直接的な宣伝。
- だが長期的に最も強力なのは **ユーザーの制作動画が自然と SYNESTA の宣伝になる** 構造。
  「制作画面 = 作品」という製品特性が、このループの基盤となる。
- 鑑賞者層 (§3.3) は「いいね・RT するが使わない」人々であり、
  制作者の「共有したい」動機を支えるエコシステムの重要な一部。

## 9. Open Questions

> 以下は現時点で確定していない検討事項。living document として随時更新する。

- **ネーミング**: SYNESTA は仮名。OS 名とすべきか、内部 DAW アプリ名とすべきか。
  OS が音楽制作以外 (LiveCoding, 画像編集, アニメーション, ゲーム) にも
  拡張される予定であるため、音楽を連想させる名前を OS 名にすべきか要検討。
- **公開形態**: building in public として制作過程を共有しつつ、
  どの段階で "正式リリース" とするかの定義。
- **コミュニティ形成**: p5.js のように、ブラウザネイティブであることを活かした
  作品共有文化をどう設計するか。
  - **バイラルループ設計**: ユーザーが SYNESTA で制作した動画を SNS に投稿すると、
    それ自体が SYNESTA の宣伝になる構造をどう強化するか
    (透かし? 共有ボタン? ハッシュタグ自動付与?)。
  - **改造・カスタマイズの可否**: 「誰でも改造可能」路線は拡散に有利だが、
    「美しさの妥協なし」原則と、改造版が SYNESTA ブランドで拡散されるリスクとの
    バランスをどう取るか。PICO-8 はゲームプラットフォームだから成立した戦略であり、
    美学が製品の核である SYNESTA には異なる判断が必要かもしれない。
  - **鑑賞者層の戦略的位置づけ**: 「使わないが観る・RT する」層は
    エンゲージメントの大部分を担い、制作者の共有動機を支える。
    この層へのリーチを意識的に設計すべきか。
- **マネタイズ**: 現時点では未定。
  - 注意: 有料で制約を緩和する方向 (パレット追加等) は、
    「制約の維持」という設計原則と哲学的矛盾を招くリスクがある。
