# MIDI エディター (ROLL) — v1 スペック

> 音楽アプリ再設計の第 2 弾。単一パターン／単一ボイスの **フレーズエディタ**。
> SYNTH ([synth v1](../js/app/synth/synth.js)) に続く独立アプリで、SYNESTA の
> モノリス的破綻への回答として「音符と時間だけを扱う」ことに徹する。
>

> 本書は「なぜそう作るか (設計意図・非目標・境界)」を保持する。数値は特記なき限り **提案値**。

## 1. 正体 (一行定義)

**「音符 (ピッチ) と時間だけを扱う、単一パターン／単一ボイスのフレーズエディタ」**

SYNESTA を破綻させたのはスコープ (音色・音量・複数トラック・アレンジ・録音・保存を 1 窓に
統合) だった。ROLL はその反省を制度化し、**1 個のノート列 (クリップ) を編集する作業台**に
徹する。音色は SYNTH、ミキシングは将来の MIXER、複数クリップの配置は将来の SEQUENCER に
委ねる。**このアプリにトラック選択列・音色パネル・ミキサーが生えたら黄信号。**

## 2. スコープ

### v1 に含む

- **単一パターングリッド編集** — ステップ量子化のノート配置／削除／長さ変更。
- **2 軸スクロール + ズーム** — 128 ピッチ × 最大 128 ステップの仮想グリッドを 360px 窓で巡る。
- **左端ピアノ鍵盤列** — 縦向き 1-bit 鍵盤 (SYNTH `keyboard.js` の描画ロジックを転用)。
- **ステップ／小節ルーラー** — 上端に拍・小節の目盛り。
- **可聴編集** — 配置時オーディション (置いた瞬間に鳴る) + プレイヘッド付きループ再生。
- **Undo / Redo** — エディタの必須装備 (スコープ膨張ではなく table stakes)。
- **VFS 保存／読込** — フラットな JSON クリップとして永続化 (§4)。

### v1 に含まない (非目標 = 脱 SYNESTA の規律)

複数トラック / ミキサー / 音色・ADSR / CC・オートメーション / ベロシティ編集 (固定) /
`.mid` コーデック本体 (モデルは互換形状にするが入出力は fast-follow) / メトロノーム /
スナップ分解能切替 (16 分固定) / 選択範囲のラバーバンド・クローンドラッグ (旧実装の
品質課題源。§5 の当たり判定を参照)。

## 3. コア画面 UI

```
        ┌──────── ルーラー (拍/小節) ────────┐
  ┌──┬─────────────────────────────────────┐
  │C5│    ██  .   .   .   .   .   .   .    │  ← グリッド (ディザ線)
  │B4│ .  .   ██████  .   .   .   .   .    │     ノート = 塗り矩形
 鍵│A4│ ██ .   .   .   .   .   .   .   .    │
 盤│G4│ .   .   .   ██  .   .   .   .   .   │
 列│..│               ▏ ← プレイヘッド (縦反転バー)
  └──┴─────────────────────────────────────┘
      └ 横=時間 (step) / 縦=ピッチ (MIDI) の 2 軸スクロール ┘
```

- **セル量子化**: ノートは常にステップ (既定 16 分 = steps/beat 4) にスナップ。時間は
  1-bit がピクセルを離散化するのと同じ精神で離散化する (連続 = "アナログと見分けが
  つかないデジタル" = PRODUCT_BRIEF が否定した当のもの)。
- **鍵盤列**: 白鍵/黒鍵で音高を可視化。オクターブ境界を強調。縦スクロールに追従。
- **ルーラー**: 小節頭を濃い線、拍を中間、ステップを淡いディザで区別。

## 4. データモデル / 永続化

`music-apps-direction` の既定に従い、**共有ノート/MIDI モデルは `core/` に置く**
(app ローカルにしない)。3 つの音楽アプリ (SYNTH / ROLL / 将来の Viewer・SEQUENCER) が
同じモデルを中心に回る。

- **配置**: `core/` に新モジュール (例 `core/clip.js` または `core/midi_model.js`)。
- **v1 形状 (最小・フラット)**:

  ```
  clip = {
    steps,                       // パターン長 (ステップ数、実効ループ終端)
    ppq  or stepsPerBeat,        // 時間解像度 (.mid 互換のため tick 換算可能に)
    notes: [ { pitch, start, len } ]   // pitch=MIDI番号, start/len=ステップ
  }
  ```

- **`.mid` 互換性**: velocity は v1 では扱わない (固定) が、**モデルの形は Standard MIDI
  File に写せる粒度** (pitch = MIDI 番号、時間 = tick 換算可能) で設計し、後の `.mid`
  入出力でモデルを作り直さなくて済むようにする。これは既定事項 (後付けにしない)。
- **v1 の永続化**: ネイティブ JSON を VFS に保存 (`core/vfs.js` の `writeFile` /
  `readFile`)。マルチトラック・CC・オートメーションは形式にすら入れない。
- **`.mid` コーデック本体**: fast-follow (v1.1)。モデルが互換形状なので追加時に再設計不要。
  バイナリ入出力は `writeFileBinary` / `readFileBinary` を使う。

## 5. 操作モデル (ステップグリッド量子化)

旧 `piano_roll.js` (40KB) の品質課題の正体は、**低解像度・1-bit で当たり判定 2〜3 DOT
しかないノート端に、自由ドラッグ 6 モード (移動/左右リサイズ/ラバーバンド/クローン/
モノフォニック切詰め) を載せた**こと。セル量子化はこの複雑さの大半を「作らない」で消す。

| 操作 | 挙動 |
|---|---|
| 空セルをクリック | 1 ステップのノートを配置 (即オーディション) |
| ノート上をクリック | 削除 (トグル) |
| ノートを横ドラッグ | 長さをセル単位で伸縮 (端掴みの精密ドラッグは廃止) |
| ホイール | 縦/横スクロール (WM が消費) |
| Ctrl + ホイール | ズーム (WM がアプリへ透過。セル固定メトリックを変更) |
| Undo / Redo | 編集の巻き戻し (Ctrl+Z / Ctrl+Shift+Z 等) |
| 再生/停止 | パターンをループ再生、プレイヘッド表示 |

- 毎フレーム reflow はしない (fixed-size + scroll に統一)。ズームは明示コマンド時のみ
  セル寸法を変える。

## 6. 1-bit 表現規約

2 色のみ。色分け (トラック色・選択色・ベロシティ色) は封じられ、表現は
**塗り／枠／ディザ／反転／位置／動き**だけ。

- ノート = 塗り矩形。選択 = 反転 or 二重枠。グリッド線 = ディザ／点線。小節頭 = 濃い線。
- プレイヘッド = 縦の反転バー。再生中に通過したノートを反転で光らせる
  (= 「制作画面 = 作品」の映え素材。差別化 #1)。
- 無効状態はグレーアウト不可 → show/hide で表現 (`one-bit-no-grayout`)。

## 7. 音源

- **内蔵デフォルト PolySynth を 1 個**持つ (`core/audio.js` の `createPolySynth()` を遅延
  生成、SYNTH と同じ前例)。**音色コントロールは持たない** (ADSR/波形を載せた瞬間 SYNESTA
  へ逆戻り)。音作りは SYNTH の領分、将来の連携で解決する。
- 2 段の可聴性: (a) 配置時オーディション、(b) プレイヘッド再生。どちらも PolySynth 経由。
- 無音エディタは「即触れる体験」(PRODUCT_BRIEF §5-4) に反するため採らない。

## 8. レイアウト制約

- **360px 既定幅で成立**させる (`ui-360-default-width`)。鍵盤列 (~24–32 DOT) +
  スクロール chrome を除いた実効グリッドは狭いので、横スクロール前提。
- **2 軸スクロールは署名かつ必然** (`scrollbars-always-functional`)。128 行グリッドは
  NOTEPAD 型 `wmAttachScroll` (行/桁単位のアプリ管理スクロール + 仮想化描画) が適する。
  Ctrl+ホイールがアプリへ透過する仕様がズームに嵌る。
- 上限グリッド = steps/beat 4 · beats/bar 4 · 128 ステップ (= 8 小節)。config に既存の
  `PIANO_ROLL_*` 定数があるが、新アプリ用に整理/改名は実装時判断。

## 9. テンポ / パターン長

- **テンポ**: BPM を NumberBox 1 個 (既定 120 = `DEFAULT_BPM`、範囲 `BPM_MIN`..`BPM_MAX`)。
  ノートはステップで保持し、BPM は再生時パラメータ (Tessera の loop 相対思想に近い分離)。
- **パターン長**: 上限 128 ステップのグリッド内で、ループ終端マーカーが実効長を決める。
- **グリッド分解能**: v1 は 16 分固定。1/8・1/32・三連のスナップ切替は将来。

## 10. 配置 (ファイル構成案)

`js/app/synth/` に倣い `js/app/roll/` に分離:

- `roll.js` — ウィンドウ登録 (`wmRegister`) + パネル配線 + 再生/保存/読込。
- グリッド編集・描画・ヒットテストのモジュール (旧 `piano_roll.js` は**参照も流用も
  しない**。ゼロベース設計。`ui-design-crap` に従い CRAP で構成)。
- 鍵盤列は SYNTH `keyboard.js` の描画ロジックを縦向きに転用/共有。
- **共有モデルは `core/`** (§4)。VFS 保存は `core/vfs.js`。音源は `core/audio.js`。
  MIDI 鍵盤でのステップ入力を将来足すなら `core/midi_input.js`。

## 11. 命名

**`ROLL`** (暫定)。理由: SYNTH / PAINT / NOTEPAD と同じ **機能直球** の系譜に属し、
コア UI メタファ (ピアノロール) を直接名指す短い大文字名。将来の兄弟 (MIXER / SAMPLER /
SEQUENCER) と衝突せず、`MIDI` 単独 (デバイス入力と紛らわしい) より app 名として自然。
`APP_NAME` 文字列 1 個なので変更は容易。代替: `MIDI`。

## 12. 「MIDI ビューアー」との棲み分け

方針では Viewer と Editor は別アプリ。**Editor (ROLL) = 実用的な作業台 (機能優先、多少
地味でよい)** / **Viewer = 再生専用の映えショーケース (ノートが流れる、読み取り専用、
鑑賞層 §3.3 向け)**。両者はクリップモデル (§4) と描画プリミティブを共有。こう割ることで
Editor は「映え」の重責を負わず機能に集中でき、Viewer が見せ場を担う。
