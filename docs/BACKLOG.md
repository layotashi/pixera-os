## Critical Path — 最重要機能 (コードベース分析で追加)

> コードベース分析の結果、**BACKLOG 既存項目よりも優先度が高い構造的課題** を発見。
> アプリと VFS の断絶は仮想 OS としてのコア体験を損なっている。

- ~~`[P0]` **アプリ–VFS 連携の実装 (Notepad / Paint のファイル保存・読み込み)**~~ ✅ 実装済み
  — Notepad・Paint ともに VFS 連携を実装。
  Alt+N (新規) / Ctrl+O (開く) / Ctrl+S (上書き保存) / Ctrl+Shift+S (名前を付けて保存)。
  未保存変更時の確認ダイアログ、タイトルバーにダーティマーク `*` 表示。
  — 関連: Explorer からのファイルダブルクリックで対応アプリで開く機能 (ファイル関連付け)。

- `[P2]` **Settings / Capture の毎フレーム再計算の最適化**
  — `settings.js` の `refreshBgDropDown` 等、`capture.js` の `refreshTargetItems()` が
  `onDraw` 内で毎フレーム呼ばれている。ダーティフラグ方式でイベント駆動に変更する。

- `[P3]` **kernel.js の未使用 `frameCount` 変数の削除**
  — `frameCount` が宣言・インクリメントされているが、どこからも参照されていない。

- `[P2]` **見た目の区別が分かり辛いウィジェットのデザイン改善**
  — Button 系・TextBox・NumberBox 等がすべて角丸矩形で視覚的に区別しづらい。
  ボーダー・インセット・テクスチャ等で各ウィジェットの "アフォーダンス" を明確にする。

- `[P3]` **ウィジェット・レイアウトシステムの整理**
  — 現在の手動配置ベースのレイアウトをより構造的にする。
  既存フレームワーク (Bootstrap 等) の Grid/Flexbox 的な概念を参考にする案あり。
  `layout.js` の拡充、または宣言的レイアウト API の設計を検討。
  **部分対応済み:** gap の不変条件 (MIN_GAP) を `ui_constants.js` / `layout.js` に導入。
  Box コンストラクタで gap < MIN_GAP を自動クランプ。
  各アプリのローカル gap 定数 (ROW_GAP / LINE_GAP 等) を廃止し、
  標準 GAP に統一済み。

- `[P3]` **BayerPicker の 2×2 マトリクス対応**
  — 現在 4×4 / 8×8 に対応。2×2 は粗いパターンだが、ローファイ感が出るので選択肢としてあり。

- `[P3]` **文字列選択・コピペ機能の拡充**
  — 現状 TextArea / TextBox 等の入力系ウィジェットではコピペ可能だが、
  Label 等の非入力要素のテキストは選択・コピーできない。
  ウィンドウ内のテキストをドラッグ選択→クリップボードコピーできると便利。
  仕様は要検討: 選択範囲の視覚的フィードバック (反転表示)、
  選択可能な要素の範囲 (Label のみ? フッター? ステータスバー?)、
  ウィジェット横断の選択を許可するか等。

### GenArt (生成的アート)

— `genart.js` (9 アルゴリズムによる生成的アート) を実装済み。

**2026-02-22 修正完了:**

- **REACT**: 格子サイズ (`RD_W`/`RD_H`) を `artWidth`/`artHeight` から動的計算に変更。解像度変更に完全対応。
- **ATTRACT**: seed 由来の ±0.02 パラメータ摂動を導入。seed ごとに異なるアトラクタ形状を生成。
- **ATTRACT>NEBULA**: 固定点に収束するパラメータを非収束値に差替え。
- **LSYS**: 角度 ±8%・線長 ±12% の seed 由来ジッターを導入。RNG 状態保存/復元で 2 パスの座標一致を保証。
- **SPIRAL**: 全体回転・位置スケール・ドットサイズ・log 巻数/成長率・helix 周波数を seed から導出。
- **AUTOMATA (center)**: 初期行に確率 1〜3% のランダムビットを混入し、対称性を崩壊させる。

## Visual / Design — ビジュアル・デザイン

- `[P2]` **画面の明るさ (Brightness) 調整機能**
  — Settings から画面の明るさを 0–100% で調整できるようにする。
  物理モニターの輝度は SYNESTA (ブラウザ) から直接制御できないため、
  配色設定の RGB 値に明るさ係数を乗算した色を VRAM 描画時に適用する方式が有力。
  実装方針の検討:
  - パレットの前景色・背景色それぞれに `brightness (0.0–1.0)` を乗算して
    実効 RGB を算出する (例: `r * brightness, g * brightness, b * brightness`)
  - 0% で完全黒、100% で元の配色どおり
  - Canvas の `globalAlpha` や CSS `filter: brightness()` で代替する案もあるが、
    VRAM ベースの描画パイプラインとの整合性を考慮する必要がある
  - Settings UI にスライダー (または NumberBox) を配置
    レトロ OS 的な「モニター調整」の雰囲気が出せると世界観にも合致する。

- `[P1]` **Display Profile の導入と表示効果の分類整理**
  — 現状の DISPLAY_TUNING は LCD 由来 (Pixel Grid) と CRT 由来
  (Diagonal scanline / Vignette / Glow) のエフェクトを「レトロ風」という
  漠然としたまとまりで混在させており、物理的に成立しない組合せが
  デフォルトで発火している (例: Game Boy LCD パレット + CRT 走査線)。
  パレット側は `origin` フィールドで既に表示テクノロジー
  (CRT 蛍光体 / LCD / Plasma / 紙 等) を暗に宣言しているが、
  エフェクト側には対応する分類が無い。

  PRODUCT_BRIEF §1「空想のマシン」は不整合を許す前提ではあるが、
  §5.3「美しさの妥協なし」と緊張する。「空想」と「ご都合主義」は別物であり、
  デフォルト状態の物理的整合性を整える価値がある。

  **改修方針**:
  - DISPLAY_TUNING の最上段に **Display Profile** セレクタを追加
    (LCD / CRT / PLASMA / PAPER / CUSTOM)
  - プロファイル切替で対応するエフェクト一式が一括 ON/OFF + 推奨初期値が適用される
  - UI はプロファイル系統ごとにセクション見出しを付ける
    (`LCD EFFECTS` / `CRT EFFECTS` / `UNIVERSAL`)
  - パレット選択時に推奨プロファイルをヒント表示する
    (強制はしない — PRODUCT_BRIEF §5.1)

  **プロファイル ↔ パレット対応案**:
  - **LCD**: Pixel Grid ON, Glow ON (微), 他 OFF → dmg, pocket_lcd, blue_lcd, el_teal
  - **CRT**: Pixel Grid OFF, Glow ON, Diagonal ON, Vignette ON, Noise 微 → p1_green, p3_amber, p4_white, p7_blue, macintosh, vectrex
  - **PLASMA**: Pixel Grid OFF, Glow 強, 他 OFF → plato
  - **PAPER**: 全エフェクト OFF (または紙質感ノイズのみ) → blueprint, thermal, e_ink
  - **CUSTOM**: 個別調整 (既存ユーザーの値はここに保存される)

  **未確定事項**:
  - PAPER プロファイルで Noise を「紙の質感」に流用するか別エフェクトにするか
  - パレットとプロファイルの連動の強さ (自動切替 / 推奨提示のみ / 完全独立)
  - 新プロファイル候補: VFD (蛍光表示管), OLED, Vector CRT (Vectrex 専用) 等
  - 既存ユーザーの体験を壊さないためのマイグレーション
    (現状値を "CUSTOM" として保存)

  **前提作業**: Pixel Grid マスタートグル
  (ジオメトリ層 = LCD ドットマトリクスと、フォト層 = 積層エフェクトの責務分離)。
  単独でも有用なため別エントリとして先行実装する。

- `[P1]` **各アプリのアイコン作成**
  — ウィンドウヘッダ・デスクトップアイコン・タスクバー等に表示するアプリ固有のアイコン。
  1-bit 美学に沿ったピクセルアートとして制作する。
  対象: Studio, Paint, Notepad, Settings, Explorer, Life Game, Capture,
  About, Breakout, GenArt, Graze, (Console)

- `[P2]` **アプリ起動時のローディング演出の追加**
  — 現在アプリ (STUDIO, NOTEPAD, PAINT 等) はデスクトップアイコンのダブルクリックや
  コンテキストメニューから起動すると即座にウィンドウが表示される。
  これはレトロ OS らしくない。意図的に「起動に時間がかかっている」ように見せる演出を
  挟むことで、空想マシンとしてのリアリティとロールプレイ感を高める。
  実装アプローチの候補:
  - **スプラッシュスクリーン方式**: ブート時の `splash.js` と同様に、
    アプリ固有のスプラッシュ画面 (ロゴ + プログレスバー等) を一定時間表示してから本体を表示
  - **ウィンドウ内ローディング**: ウィンドウ枠は即座に表示するが、
    コンテンツ領域にローディングアニメーション (砂時計・プログレスバー・ディザパターン等) を
    表示し、一定時間後にアプリ本体の UI に切り替える
  - **カーソル変化**: 起動中はカーソルを砂時計に変更 (「カーソル切り替え」項目と連動)
    アプリごとに演出の長さや内容を変えても面白い
    (例: STUDIO は重いので長め、NOTEPAD は軽いので短め)。
    過度な待ち時間はストレスになるため、演出時間のバランス調整が重要。

- `[P2]` **ディザグラデーション描画プリミティブの活用**
  — `bayerGradRect` を UI 装飾に積極活用する。ウィンドウヘッダのアクセント、
  ダイアログ背景、ボタンホバー効果等に適用すると空想マシンの世界観が強まる。

- `[P2]` **カーソルの種類と見た目切り替えの充実**
  — 押下可能要素にホバーしたときのポインターカーソル、クリック中の押下カーソル、
  テキスト入力時の I-beam、待機中の砂時計等、状態に応じたカーソル切り替えを実装する。
  cursor.js のスプライトシートにバリエーションを追加。

- `[P3]` **文字装飾 (太字・下線) 対応のためのテキスト描画・フォント拡充**
  — 太字 (Bold) と下線 (Underline) をテキスト描画プリミティブレベルでサポートする。
  太字はグリフの 1px 右シフト重ね描き、または専用の Bold フォントシート追加で対応。
  下線は `drawText` に装飾フラグを追加し、ベースライン下に hline を描画。
  イタリックは低解像度 (5px 幅グリフ) では視認性が悪いため、現時点では不要と判断。
  `font.js` / `gpu.js` の API 拡張、および `assets/font.png` の追加シート作成が必要。

## Architecture — アーキテクチャ

- `[P3]` **TypeScript 移行 (V2)**
  — README に記載あり。型安全性と IDE 支援の向上を目指す。
  ビルドレスの原則との両立方法を検討 (例: 型チェックのみ、JSDoc 型注釈、等)。

- `[P3]` **開発用デモアプリの整理 (gradient_demo.js / easing_demo.js)**

  開発確認用のデモウィンドウ。動作確認完了後に削除するか、
  開発者ツールとして残すかを判断する。

- `[P2]` **テストコードの拡充**
  — vitest 導入済み (`npm test` で実行可能)。現在の単体テスト:
  `anim.test.js`, `vfs.test.js`, `dither.test.js`, `audio.test.js`,
  `layout.test.js`, `ui_helpers.test.js`, `ascii_art.test.js`
  今後の拡充方針:
  - カバレッジの低いモジュールへのテスト追加
  - ヘッドレスブラウザ (Playwright 等) での Canvas 描画スナップショット比較
  - VRAM ダンプ機能 (`vram_dump.js`) と組み合わせた画面アサーション

- `[P3]` **命名規則の "OS 的文脈" への見直し**
  — V0.1.0 時点では SYNESTA が仮想 OS であるというコンセプトが明確でなかったため、
  Web 系開発由来の命名 (識別子・ファイル名・フォルダ名等) が混在している可能性がある。
  仮想 OS としての世界観を強化するために、OS 的文脈で使われる用語・命名規則へ
  段階的に統一することを検討する。
  例: "widget" → "control"、"app" → "program" or "application"、
  "callback" → "handler" or "message"、"render" → "draw" or "paint"、等。
  まず現状の命名を棚卸しし、変更による影響範囲と優先度を整理した上で着手する。

## Ideas — アイデア・将来構想

> PRODUCT_BRIEF.md に記載の将来構想、およびまだ形になっていないアイデア。

- **LiveCoding 環境**
  — Notepad をベースに、コード実行・音声/映像生成ができるライブコーディング環境を構築。
  Console ウィンドウと連携して出力を表示。

- **アニメーション制作ツール**
  — Paint の拡張として、フレームベースのドット絵アニメーション制作機能を検討。
  オニオンスキン・タイムライン・GIF/APNG エクスポート等。
  ※ CAPTURE の GIF ループ撮影 (画面録画の GIF 化) とは別機能。
  こちらは自作ドット絵アニメーションの書き出しであり、`core/gif.js` の
  GIF エンコーダを共用できる。

- **ゲーム制作ツール (Fantasy Console 的機能)**
  — PICO-8 / TIC-80 的な、SYNESTA 内で完結するミニゲーム制作環境。
  スプライトエディタ・マップエディタ・コードエディタを統合。

- **2-bit / モノクロ＋アクセントカラー拡張**
  — PRODUCT_BRIEF に「検討対象」として記載。明暗 4 階層やアクセントカラー 1 色の追加。
  コンセプトの根幹に関わるため慎重に判断する。

- **右クリックコンテキストメニュー**
  — デスクトップやウィンドウ内での右クリックメニュー。
  ファイル操作・クリップボード・ウィンドウ操作等のショートカットを提供。

- **キーボードショートカット一覧 / カスタマイズ**
  — 現在のショートカットの一覧表示、および将来的にはユーザーカスタマイズを可能にする。

- **スプラッシュスクリーンのカスタマイズ**
  — ブート演出のバリエーション追加やユーザーによるカスタマイズ。

- **PWA 対応 (オフライン起動)**
  — Service Worker を導入し、オフラインでも使用可能にする。
  ブラウザ完結の強みを維持しつつ、インストール体験を追加。

- **ユーモア・ジョーク・イースターエッグ**
  — `HUMOR_PRINCIPLES.md` にユーモア設計原則を策定済み。
  8 つの抽象原則 + 運用ガイドライン + 評価チェックリストで構成。
  今後のジョークプログラム・イースターエッグはすべてこの原則に準拠して設計する。
  **実装済み:** `dolphin.js` (パターン B: 無害な執着を持つプログラム)、
  デスクトップ複数行ラベル。
  **今後の方向**: `HUMOR_PRINCIPLES.md` §3 に設計パターン A–D を定義済み。

- **AI 機能との統合**
  — レトロ OS 風でありながら現代的機能を内包するという SYNESTA の方針に合致する。
  「レトロな 1-bit 画面で最新 AI 技術に触れる」体験は、見た目と中身のギャップ自体が
  面白さ・驚きを生む可能性がある。想定される適用先:
  - **PianoRoll**: メロディー・コード進行・リズムパターンのサジェスト。
    数小節の入力から続きを提案、ジャンル/ムード指定での自動生成等
  - **Notepad → IDE 化**: コード補完・リファクタ提案。
    将来の LiveCoding 環境と組み合わせればかなり強力
  - **Paint**: ドット絵の自動補完・パターン生成、テキストプロンプトからの 1-bit 画像生成
  - **システムレベル**: 自然言語でのコマンド実行、ヘルプ/チュートリアルの対話的ガイド
    技術的難易度は高め。外部 API (OpenAI, local LLM 等) との通信が必要になり、
    ブラウザ完結・ゼロ依存の原則とのトレードオフが生じる。
    API キー管理、レイテンシ、オフライン時のフォールバック等も要検討。

- **文字列 → ASCII アートロゴ変換アプリの追加 / ASCII 系機能の拡充**
  — 入力文字列を ASCII アートのロゴに変換する機能。独立したウィンドウアプリとして
  提供するのが自然。Linux の `figlet` / `toilet` が同等機能の有名パッケージ。
  アプリ名は figlet を彷彿とさせつつ独自の名称を検討
  (例: "BIGLET", "TYPESET", "BANNER" 等)。
  複数フォント (Banner, Block, Slant 等) の切替、出力のクリップボードコピー、
  VFS へのテキストファイル保存等があると実用的。
  現状 SYNESTA で ASCII 文脈が活かされているのはブート時のロゴ表示 (`config.js` の
  `APP_ASCII_LOGO`) 程度。チップチューン・ドット絵と並んで ASCII アートも当然 SYNESTA の
  1-bit 美学のスコープに入るため、この方向の機能拡充は自然な展開と言える。

- **Web ブラウザアプリの実装 + SYNESTA 専用レトロ Web 圏**
  — SYNESTA 内で動作する独立ウィンドウ型の Web ブラウザ。
  現代の Web サイトは HTML/CSS/JS が高度すぎて 1-bit・低解像度環境ではまともに
  閲覧できない可能性が高い（要検証）。そこで逆転の発想:
  - **SYNESTA 専用の制限付き HTML 仕様**: 現代 HTML と互換性を持ちつつ大幅に制限した
    シンプルなマークアップ言語を定義する。テキスト・リンク・画像 (1-bit) ・リスト程度。
    CSS は最小限 or 独自スタイル、JS は不可 or サンドボックス実行等
  - **SYNESTA Web サーバ**: プロジェクト側でホスティングサーバを用意し、
    世界中の SYNESTA ユーザーが作成したシンプルな Web ページを公開・閲覧できるようにする。
    ユーザー同士で独自の「レトロ Web 文化圏」が自然発生する可能性がある
  - **世界観との親和性**: 初期インターネット (Geocities, テキストブラウザ) の雰囲気と
    SYNESTA の空想レトロマシンの世界観は非常に相性が良い。
    制作 (音楽・ドット絵) の共有プラットフォームとしても機能し得る
    技術的にはブラウザ内ブラウザ (HTML パーサ + 簡易レンダラ) の実装、
    サーバインフラの運用コスト、コンテンツモデレーション等の課題がある。

## Beta Release — ベータ公開に向けた検討事項

- `[P2]` **モーダル EXPLORER のファイル操作**
  — 現在の FileDialog (`ui/FileDialog.js`) は VfsBrowser を内蔵しているが、
  ファイルの作成・リネーム・削除はメインの Explorer ウィンドウでしか行えない。
  FileDialog 内でも最低限のファイル操作 (新規作成・リネーム・削除・移動) を可能にする。
  現代 OS のファイル選択ダイアログはほぼ例外なく基本的なファイル操作をサポートしており、
  「壁紙を設定したいだけなのに Explorer を開いて整理してからやり直す」という
  フローは世界観の没入を妨げる。
  VfsBrowser の操作 API を拡充し、FileDialog からも呼び出せるようにする方向。

- `[P2]` **壁紙デフォルト値の変更 (SOLID → IMAGE + DEFAULT.PBM 同梱)**
  — 初回起動時の壁紙を SOLID (Bayer ディザ) から IMAGE モードに変更し、
  デフォルト壁紙 (`/IMAGES/WALLPAPERS/DEFAULT.PBM`) を VFS に同梱する。
  Design Principle #2「映える画面 over 隠れた機能性」、#4「即触れる体験」の観点から、
  初見ユーザーへの引力を強化する。
  検討事項: 1-bit で「映える」壁紙のデザイン制作、複数解像度対応、VFS 初期化時の同梱方法。

- `[P3]` **ドライブレターの導入検討**
  — レトロ OS の UX として `A:` `C:` 等のドライブレター表記は没入感を高める。
  VFS ルートを `C:\` にマッピングするか、マウントポイント (`/mnt/c/`) 方式にするか、
  または表示レイヤーのみで変換する (内部パスは `/` のまま) か、3 つのアプローチがある。
  永続ストレージ (localStorage) のキーにも影響するため、慎重に検討する。
  β版時点では表示レイヤー変換 (内部 `/` → UI 表示 `C:\`) が最も安全。
