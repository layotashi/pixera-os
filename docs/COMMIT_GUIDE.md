# PIXERA OS — コミット規約

> 目的: `git log` を後から読んで「何が・なぜ変わったか」を追えること。
> 強制ではなくガイド。**迷ったら Subject 1 行だけでよい。**
> Conventional Commits の軽量版。

---

## 最短テンプレート

```
<type>(<scope>): <日本語の概要>

<body: なぜ変えたか。任意>

Note: <English, user-facing, 1 line. WELCOME に載せたい時だけ>
Co-Authored-By: ...
```

| 要素     | 言語   | 必須 | ひとこと                                   |
| -------- | ------ | ---- | ------------------------------------------ |
| `type`   | 英語   | ✅   | 変更の種類 (下の表から 1 つ)               |
| `scope`  | 英語   | ❌   | 影響範囲。1 語で言えるときだけ             |
| 概要     | 日本語 | ✅   | 50 字目安・句点なし・現在形                |
| `body`   | 日本語 | ❌   | **なぜ** 変えたか。WHAT は diff で読める   |
| `Note:`  | 英語   | ❌   | ユーザーへの告知。WELCOME 用 (後述)        |
| footer   | 英語   | ❌   | `Co-Authored-By:` 等                       |

## 3 ステップで書く

1. **type を選ぶ** — 下の表から 1 つ。
2. **概要を日本語で 1 行** — 「何が起こるか」を書く (「何を変えたか」ではない)。
3. **必要なら足す** — WHY があれば `body`、ユーザーに見せたいなら `Note:`。無ければ省略。

## type 一覧

| type       | 意味                            |
| ---------- | ------------------------------- |
| `feat`     | 新機能 (ユーザーが触れるもの)   |
| `fix`      | バグ修正 (意図通りに戻す)       |
| `refactor` | 動作を変えず構造を変える        |
| `perf`     | パフォーマンス改善のみ          |
| `docs`     | ドキュメントのみ                |
| `test`     | テストのみ                      |
| `style`    | 整形のみ (動作変更なし)         |
| `chore`    | 雑多 (アセット差替・設定ファイル等) |
| `revert`   | 直前コミットの取り消し          |

> `build` / `ci` は不要 (ビルドレス)。必要になったら `chore` に吸収。

## scope (任意)

レイヤー名かアプリ名を英語小文字で。階層は `/` 区切りで **2 段まで**。
1 語で名指しできないとき・複数レイヤーに跨るときは **省略する**。

```
✅ fix(wm): スナップ復帰時の座標ずれを修正
✅ chore: .gitignore を追加            (scope 省略)
❌ fix(wm/scrollbar/thumb): ...        (階層が深すぎ)
```

例: `wm` `ui` `audio` `kernel` `config` `splash` / `core/gpu` `core/vfs` /
`synesta` `paint` `notepad` `tessera` `settings` / `docs` `backlog`

## 概要 (Subject)

- **50 字目安**。超えたら `body` に分ける。
- **句点 (。) を付けない**。
- **現在形・命令形** (`修正する` / `追加` / `緩和`)。過去形・完了形は避ける。
- **何が起こるか** を書く。

```
❌ fix(wm): minH の計算ロジックを変更     (何を変えたか)
✅ fix(wm): scrollable の最小高さを緩和    (何が起こるか)
```

## Note: トレーラー (WELCOME のリリースノート用)

PIXERA OS の起動画面 **WELCOME** に出る「WHAT'S NEW」は、コミットの `Note:` 行を
pre-commit フックが集めて作る。**ここが唯一のソース**。日本語 Subject はそのまま残す。

### 付けるか付けないか

- **付ける**: ユーザーが気づく変化。主に `feat`、体感できる `fix`。
- **付けない**: 内部変更 (`refactor` `perf` `docs` `test` `style` `chore`) や些細な修正。
  付けなければ WELCOME に載らないだけ。**無理に付けなくてよい。**

### 書式 (WELCOME は 5x5・大文字・ASCII でしか描画できない)

- **英語で書く** — 日本語は描画できず欠落する。
- **ASCII 文字だけ** — 使えるのは英数字と `- . , ( ) / : % ! ? _ #` 等。
  em-dash `—`・中黒 `·`・スマートクォート `“ ”` は不可。
- **短く 1 行** — 小窓に 1 行で収まる目安 **40 字以内**。
  大文字化は OS が自動で行うので、普通の英文で書けばよい。
- **1 コミットに 1 つ** — 見せ場が複数あるなら、そもそもコミットを分ける。
- **ユーザー視点** — 「何ができるようになったか」。コードの説明ではない。

### 例

```
feat(paint): フラッドフィルツールを追加

Note: Paint now has a flood fill tool
```

```
fix(capture): GIF録画の早回しを修正

Note: GIF capture now records at the correct speed
```

```
❌ Note: リファクタで内部を整理        (日本語・ユーザーに無関係)
❌ Note: refactored wm.js internals    (内部変更・ユーザー視点でない)
```

## body / footer (任意)

**body** — 「diff だけでは伝わらない WHY」があるときだけ書く。
Subject から 1 行空け、72 字目安で折り返し、箇条書きは `-`。

```
fix(wm): scrollable ウィンドウの最小高さを緩和

onMeasure() の自然サイズが「初期サイズ」と「最小サイズ」の両方に使われ、
コンテンツが画面より大きいと最大化でしかスクロールできなかった。
scrollable のときは MIN_HEIGHT まで縮められるよう責務を分離する。
```

**footer** — body との間を 1 行空ける。

| footer                                           | いつ                          |
| ------------------------------------------------ | ----------------------------- |
| `Note: ...`                                       | WELCOME に告知したいとき (上記) |
| `Co-Authored-By: Claude <...>`                    | Claude が編集を担ったとき (正確な文字列は環境が指定) |
| `Refs: docs/BACKLOG.md`                           | バックログ項目に紐づくとき     |
| `Breaks: <内容>`                                  | 後方互換を壊すとき             |

## やらないこと

- ❌ **中身を語らない Subject** — `fix: 修正` / `chore: いろいろ`。単独で意味が通るように。
- ❌ **body で diff を繰り返す** — WHAT は diff で読める。書くなら WHY。
- ❌ **1 コミットに複数の意図** — `revert` 単位を「1 つの意図」に揃える。分ける。
- ❌ **過去形・完了形** — `緩和した` → `緩和`。
- ❌ **type と中身のズレ** — `feat(docs):` は `docs`、`refactor` でのバグ修正は `fix`。

## 運用メモ

- **強制しない**。hook で機械的に弾かない。判断はコミット作成者に委ねる。
- **既存コミットは書き換えない**。規約導入前のものは時代の境目として残す。
- **迷ったら Subject だけ**。完璧な body より、こまめなコミット。
